1. Agent Activity Dashboard

Where: Navigate to /agent in the app. Accessible from the main navigation menu. Contains four tabs: Actions, Automations, Relay, and Stats.

Why: Every AI action — whether detected from voice input, generated by a briefing, or spawned by an automation rule — flows through this dashboard. It is your command center for reviewing what Keryx is proposing to do, what it has already done, and rolling back anything you didn't intend.

What you can do here:

  • Review pending actions — cards appear with the proposed action, AI reasoning, confidence score, and source context
  • Approve — executes the action immediately (calendar event created, email sent, etc.)
  • Reject — dismisses the action without executing it
  • Rollback — undoes a completed action (available for calendar and email actions within the rollback window)
  • Expand / collapse — reveals full payload, AI reasoning, triggered-by text, and result data
  • Filter by category — pill filters for Calendar, Email, Reminder, People, Goals, Research, Memory, Financial, and Relay
  • Filter by status — view pending, completed, failed, or rejected actions
  • View stats — the Stats tab shows pending count, completed today, failed today, and a category breakdown chart

Action statuses and their meaning:

StatusMeaning
PendingWaiting for your approval (or auto-executing if policy = Auto)
ApprovedYou approved it; execution is queued
ExecutingCurrently being carried out
CompletedFinished successfully. Result data shown on expand.
RejectedYou dismissed it; no action taken
FailedExecution attempted but errored; error message shown on expand
CancelledExpired before being acted on

Chain visualization:

  • Child actions (spawned via Action Chaining) appear indented under their parent with a violet left border
  • A "→ spawned from [parent title]" badge is shown at the top of each child card
  • Chain depth is displayed (depth 1, 2, or 3)
  • Approving or rejecting a child action is independent of the parent — each step requires its own decision

2. AI Task Execution — All 9 Action Types

Where: Detected automatically from voice and text input at memory save time. Also generated by proactive analysis, morning briefings, and automation rules. Managed in the Agent dashboard.

Why: Keryx doesn't just record — it acts. Say something naturally and Keryx extracts the actionable intent, confirms with you (unless set to auto), and executes it. Each action type routes to the appropriate backend service.

calendar.create

Creates a calendar event on your Google Calendar or Outlook Calendar (whichever is connected).

  • Payload fields: summary, startDateTime, endDateTime, attendees (array of emails), location, description, timezone
  • Duplicate check: before creating, Keryx searches for events with the same title within ±2 hours to avoid duplicates
  • Rollback: available — deletes the created event
  • Requires: Google Calendar or Outlook connected in Settings
"Schedule a dentist appointment for Thursday at 2pm"
Creates an event titled "Dentist Appointment" on Thursday at 2pm in your local timezone

calendar.delete

Deletes an existing calendar event using its event ID from Google Calendar or Outlook.

  • Payload fields: eventId (required — the calendar event's unique ID), eventTitle (display), provider (google | outlook)
  • How the ID is found: Keryx can search your calendar for the event by title and date to get the ID first, then queue the delete action
  • Rollback: not available — deletion is permanent
"Cancel my 3pm meeting today"
Looks up the event on your calendar and proposes a delete action with the event ID pre-filled

email.send

Sends a new email via Gmail or Outlook Mail (whichever is connected).

  • Payload fields: to (email address or array), subject, body, cc (optional)
  • Rollback: available — moves the sent email to trash (best-effort)
  • Chainable: yes — calendar.create actions can chain to email.send for invite emails
  • Requires: Gmail or Outlook connected in Settings
"Email the team about the product launch update"
Composes and sends a message to the recipients Keryx detects from context

reminder.create

Creates a time-based or location-based reminder stored in Keryx.

  • Payload fields: title, content, triggerTime (ISO datetime), triggerType (time | location), location (for location-based)
  • Delivery: push notification at trigger time (requires push notifications enabled)
  • Rollback: available — deletes the reminder
"Remind me to submit the report next Monday at 9am"
Creates a reminder with a push notification queued for Monday 9am in your local timezone

people.note

Adds a note to a person's contact record in Keryx's People system.

  • Payload fields: personName, personId (if known), note (the text to add)
  • How person matching works: fuzzy name match against your People database; creates a new contact if no match found
  • Chainable: yes — can be chained from calendar.create or email.send
  • Rollback: not available
"Note that Sarah mentioned she's relocating to Denver in June"
Adds "mentioned she's relocating to Denver in June" as a timestamped note on Sarah's contact record

web.search

Executes a live web search via the Tavily AI Search API and returns a summary with source links.

  • Payload fields: query (the search string)
  • Result: AI-synthesized summary + list of source URLs stored in the action's resultData
  • Rollback: not applicable
  • Requires: TAVILY_API_KEY configured
"Search for the best standing desk options under $500"
Runs a web search and returns a summary with links directly in the Agent dashboard result card

memory.create

Explicitly logs a new memory entry through the action system rather than the main input flow.

  • Payload fields: memoryText, topicTag (optional, AI-derived if omitted), mood (optional), importance (1–10)
  • Use case: automation rules that auto-log entries, or chained memories from other actions
  • Rollback: not available
"Log that I finished the Q1 financial review today"
Creates a memory entry with AI-extracted topic, mood, and importance from the text

goal.update

Updates the progress percentage on one of your tracked goals.

  • Payload fields: goalId, goalTitle, newProgress (0–100)
  • How goal matching works: fuzzy title match against your Goals list; the most recently active goal with a matching name is selected
  • Rollback: not available
"Mark my reading goal at 60% complete"
Updates the progress bar on the matching goal to 60% and logs the change

financial.alert

Surfaces a financial spending pattern or budget concern derived from your Plaid transaction data.

  • Payload fields: title, description, category, amount, period
  • Source: proactive analysis engine examines recent transactions for anomalies or high spending categories
  • Requires: Plaid connected with at least one linked bank account
  • Rollback: not applicable
"Alert me if my dining spending is unusually high this month"
Creates a financial.alert action when the proactive engine detects above-average dining expenses
Not yet supported

calendar.update (editing existing events) and email.reply (replying to existing emails) are not yet implemented. These are planned for a future release.

3. Per-Action Policies (Auto / Confirm / Disabled)

Where: Settings → AI Task Execution section. Each action type has its own policy selector.

Why: Different actions carry different risk levels. You might want calendar events to always be confirmed, but routine reminders to execute automatically. The policy system gives you fine-grained control without requiring you to approve every low-stakes action.

Three policy modes:

PolicyBehaviorBest for
Confirm (default) Action appears as a pending card in the Agent dashboard. You approve or reject. A push notification is sent if push is enabled. High-stakes actions: calendar events, emails
Auto Action executes immediately without waiting for approval. Still logged in the Agent dashboard as "completed." Low-stakes actions: reminders, people notes, memory logs
Disabled Keryx never proposes or executes this action type, even if it detects intent in your input. Action types you never want Keryx to act on

How it works:

  • Policies stored in the settings table per user as a JSONB map of action type → policy string
  • When an action is detected, the policy is checked before creating the action record
  • Auto-policy actions execute synchronously during the action detection pipeline; their results appear immediately in the dashboard
  • Confirm-policy actions are saved with status "pending" and trigger a push notification if push is configured
  • Disabled-policy actions are silently dropped — no record is created

Key files:

server/ai-actions-service.ts, client/src/pages/settings.tsx (AI Task Execution section)

4. Action Chaining

Where: Happens automatically after a chainable action completes. Child actions appear in the Agent dashboard indented under their parent. Toggle in Settings → AI Task Execution → Allow Action Chaining.

Why: Real-world tasks rarely stop at one step. After creating a calendar event, you probably want to send a confirmation email. After sending an email to a new contact, you probably want to add a note to their profile. Action chaining detects these natural follow-ups so you don't have to remember to do them — and still keeps you in control with a one-tap approve at each step.

How it activates:

  • A chainable parent action completes successfully
  • The allowActionChaining setting is true (default)
  • GPT analyzes the completed action's context and source text for a natural single follow-up
  • A follow-up action is only proposed if GPT confidence ≥ 0.75
  • If the chain depth would exceed 3, no child is spawned

Chainable action types — both parent and child actions must be one of:

  • calendar.create — may chain to an invite email (email.send) or a contact note (people.note)
  • email.send — may chain to a contact note (people.note) or another email follow-up
  • people.note — may chain to a related calendar event (calendar.create) or a follow-up email (email.send)

Important: The server enforces a hard whitelist — child actions must be one of the three chainable types above. GPT cannot propose a child action of type reminder.create, goal.update, or any other type; those would be rejected at the server level.

Chain depth limits:

DepthDescriptionExample
0Root / parent actioncalendar.create (schedule dentist)
1First childemail.send (send a confirmation email)
2Second childpeople.note (note that the appointment was scheduled)
3 maxMaximum depth — no further spawning–

How it works technically:

  • On action completion, maybeSpawnChainedAction(userId, completedAction) is called
  • GPT is given the parent action details, source text, and action type to detect one follow-up
  • If a follow-up is detected, a new AI action record is created with parentActionId pointing to the parent
  • chainDepth is set to parent.chainDepth + 1
  • The child's sourceText is prefixed: "Chained from: [parent title]"
  • The child inherits the parent's action policy (so auto-actions chain to auto-children)

Agent dashboard visualization:

  • Child actions render indented with a violet left border and a GitBranch icon
  • "→ spawned from [parent title]" badge appears at the top of the child card
  • Chain depth indicator: "· depth 1", "· depth 2", etc.
  • Each child card has its own independent Approve / Reject buttons

Key files:

server/ai-actions-service.ts (maybeSpawnChainedAction, CHAINABLE_ACTION_TYPES, MAX_CHAIN_DEPTH), shared/schema.ts (parentActionId, chainDepth columns), client/src/pages/agent.tsx (renderNode recursive UI)

Active by default Toggle: Settings → Allow Action Chaining

5. Automation Rules Engine

Where: Agent page → Automations tab. Rule management: Create, toggle, test, delete. API at GET/POST/PATCH/DELETE /api/automation/rules.

Why: Keryx is reactive — it responds to what you tell it. The Automation Rules Engine makes it proactive — it responds to things that happen in your life. Log a memory that mentions stress? A rule can immediately create a check-in reminder. Mood drops below 4? A rule can fire off a motivating notification. These run in the background without interrupting your flow.

Rule anatomy:

  • Name — a human-readable label for the rule
  • Trigger — what event starts the rule
  • Conditions (optional) — filter criteria the event must match (e.g. mood below 4, keyword = "stressed")
  • Action — what to do when the rule fires
  • Action payload — parameters for the action (supports {{variable}} interpolation)
  • Max runs per day — safety limit (default: 3 per rule per day)
  • Enabled / Disabled — toggle without deleting

All 10 trigger types:

TriggerFires when…Context variables available
memory.loggedAny memory is saved{{memoryContent}}, {{mood}}, {{personName}}
mood.droppedMood score falls below a threshold{{mood}}, {{memoryContent}}
mood.spikedMood score rises above a threshold{{mood}}
person.mentionedA specific person is detected in a memory{{personName}}, {{memoryContent}}
keyword.detectedA specific word appears in memory text{{keyword}}, {{memoryContent}}
reminder.dueA reminder reaches its trigger time{{reminderContent}}
briefing.generatedA morning briefing is created{{memoryContent}} (briefing text)
goal.updatedAny goal's progress changes{{goalTitle}}, {{progress}}
daily.scheduleDaily cron at a configured hour–
action.completedA specific action type finishes successfully{{actionTitle}}

All 5 action types:

ActionWhat it doesKey payload fields
send.notificationSends a push notification to all your devicestitle, body, url
create.reminderCreates a time-based remindercontent, minutesFromNow or dueAt
create.ai_actionQueues a new AI action for the Agent dashboardactionType, title, description, actionData
log.memoryAutomatically saves a memory entry with AI metadatacontent (supports interpolation)
relay.outboundQueues an outbound relay message for approvaldestination, message, label

Condition fields you can use:

  • moodBelow: 4 — only fire if mood score < 4
  • moodAbove: 8 — only fire if mood score > 8
  • keyword: "stress" — word-boundary match: "stress" catches "stressed", "stressful" but NOT "distress"
  • aiSentiment: "negative" — AI-derived sentiment: positive, neutral, or negative. Sourced from the AI mood label (e.g. "stressed" → negative, "happy" → positive) with numeric fallback. More intent-aware than a raw moodBelow threshold.
  • aiTopic: "Health" — preferred alias: matches the AI-assigned topic tag (same data as topic, explicit name for rules using memory.logged trigger)
  • topic: "Health" — legacy alias for aiTopic; supported for backward compat
  • personName: "Sarah" — only fire if Sarah is mentioned
  • progressAbove: 75 — only fire for goal updates above 75%
  • progressBelow: 20 — only fire for goal updates below 20%
  • atHour: 7 — only fire the daily schedule rule at 7am local time
  • actionType: "calendar.create" — only fire for completed calendar create actions

Tip: aiSentiment and topic are AI-extracted — they match intent, not just raw text. keyword now uses word-boundary matching, so it handles inflections automatically.

Template interpolation example:

Action payload:
{
  "content": "Reached out to {{personName}} — {{memoryContent}}"
}

When the rule fires (person.mentioned, personName = "Marcus"):
→ "Reached out to Marcus — Had a great coffee chat about the project"

How it works technically:

  • Rules stored in automation_rules table; conditions and action payload as JSONB
  • Every major event in the app calls fireTrigger(userId, triggerType, context)
  • Engine fetches all enabled rules for that user + trigger; evaluates conditions; checks daily run limit
  • Actions execute in a setImmediate callback — non-blocking, fire-and-forget with isolated error handling
  • Each execution records success/failure + timestamp on the rule's last-run fields
  • Test button: immediately fires the rule action regardless of conditions/limits (for debugging)

Key files:

server/automation-engine.ts, client/src/pages/agent.tsx (Automations tab), server/routes.ts (/api/automation/rules/*)

6. Universal Relay API (Inbound & Outbound)

Where: Inbound: POST /api/relay/inbound. Outbound: destinations configured in Settings. Event log and destination management: Agent page → Relay tab.

Why: Acts as a universal message bus. Any external system — your Android phone, Meta glasses, an SMS gateway, a custom script — can push data into Keryx without a browser session. Keryx can also push back out, relaying your morning briefing, action results, or custom messages to any HTTP endpoint you configure. This powers the Android Bridge and enables third-party integrations.

Inbound Relay

Endpoint: POST /api/relay/inbound

Auth: X-API-Key: <your-relay-api-key> — no session cookie required

Supported payload types:

TypeWhat it does
smsStores the message in the Keryx Messages system (visible in the Messages page). Fan-out to all enabled inbound destinations.
commandRuns AI action detection on the command text. Any detected actions are queued for approval in the Agent dashboard.
eventLogs a custom event to the relay event table. Fan-out to configured destinations.
POST /api/relay/inbound
X-API-Key: your-relay-api-key-here
Content-Type: application/json

{
  "type": "sms",
  "source": "android-bridge",
  "payload": {
    "from": "+15551234567",
    "body": "Dinner at 7pm tonight?",
    "timestamp": "2026-04-13T19:00:00Z"
  }
}

Getting your Relay API key:

  • Settings → scroll to the Relay section
  • Your API key is displayed there (generate a new one if needed)
  • The Android Bridge app uses this same key automatically once you paste it in during setup

Outbound Relay

Configure destinations in Settings → Relay → "Add Destination". Each destination has:

  • URL — the HTTP endpoint to POST to
  • Name — a label for the destination
  • Enabled — toggle to pause without deleting
  • Outbound enabled — whether this destination receives outbound messages from Keryx
  • Format — json (full envelope) or text (plain string)
  • Briefing relay — if enabled, your morning briefing text is automatically POSTed to this destination after generation

Outbound is triggered by:

  • Approving a RELAY_OUTBOUND AI action in the Agent dashboard
  • An automation rule firing the relay.outbound action type
  • Morning briefing generation (if outboundBriefingRelay = true on a destination)

Event Log

Every inbound relay event is stored per-user and visible in the Agent page → Relay tab. Columns shown: timestamp, source, type, payload preview, delivery status. Useful for debugging Android Bridge connectivity and verifying that external systems are reaching Keryx successfully.

Session-authenticated test endpoint

POST /api/relay/test — requires a browser session (no API key). Use this to test relay routing from the web app without needing the API key. Available in Settings → Relay section.

Key files:

server/routes.ts (/api/relay/*), client/src/pages/agent.tsx (Relay tab), client/src/components/settings/relay-settings-card.tsx

Active Auth: X-API-Key (inbound) / Session (test)
