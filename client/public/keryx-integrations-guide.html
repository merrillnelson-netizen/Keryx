1. OpenAI (GPT + Embeddings)

Where: Used across the entire backend. The AI engine behind nearly every smart feature in Keryx.

Why: Provides the intelligence layer. Without OpenAI, Keryx would be a simple note-taking app. It turns free-form voice or text into structured, searchable, actionable data.

When it activates:

  • Every time you save a memory (extracts topics, mood, people, importance, categories)
  • Every search query (generates embeddings for semantic matching)
  • Morning briefings (synthesizes recent data into a personalized summary)
  • Pattern detection (spots recurring themes in your memories)
  • Goal analysis and milestone suggestions
  • Thematic insights and deep pattern analysis with Q&A
  • Intent detection (determines if input is a log or a search query)
  • AI action detection (spots calendar events, emails, reminders in your input)
  • People duplicate detection and AI-powered people search
  • Financial query answering (when Plaid is connected)
  • Personal news feed generation
  • Contextual discoveries (generates search queries from your data)
  • Message conversation analysis (topics, mood, importance)
  • Ideas workspace AI assistance
  • Personal insights generation

How it works:

  • Uses GPT-4o-mini for fast, cost-effective processing of most tasks
  • Uses text-embedding-3-small for generating vector embeddings (1536 dimensions)
  • Connects via Replit AI Integration (preferred) or direct OpenAI API key as fallback
  • Rate-limited: 15 requests per minute per user on AI routes
  • 30-minute TTL cache for briefings and alerts to avoid redundant calls
  • 30-second timeout with 2 automatic retries

Key files:

server/ai-service.ts, server/message-ai-service.ts, server/ai-actions-service.ts, server/contextual-discoveries-service.ts

Active Secrets: OPENAI_API_KEY

2. Google Calendar

Where: Settings page (connection toggle), memory creation flow, morning briefings, AI actions, dashboard.

Why: Automatically links your memories to calendar events. If you log "the team meeting went well," Keryx checks your calendar and attaches the actual meeting details. Also lets AI create calendar events from natural language.

When it activates:

  • When saving a memory: Searches for calendar events happening around the same time and auto-links matching ones
  • During morning briefings: Pulls today's schedule into the briefing summary
  • AI action execution: Creates calendar events when you say things like "schedule a dentist appointment for Thursday at 2pm"
  • Contextual discoveries: Calendar events inform search queries for relevant content
  • Personal insights: Calendar data contributes to pattern analysis
  • Duplicate event checking before creating new ones

How it works:

  • Self-managed OAuth 2.0 — user authenticates via Google OAuth flow in Settings; tokens stored securely in database
  • Tokens automatically refreshed using GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET
  • Uses Google Calendar API v3 via googleapis library
  • Access tokens cached for 30 seconds, refreshed 5 minutes before expiry
  • Searches events within +/- 2 hours of memory timestamp for auto-linking
  • Handles both all-day events and timed events correctly across timezones
  • Connection status checked dynamically; gracefully degrades when disconnected
  • Required: Google Calendar API must be enabled in your Google Cloud Console project

Key files:

server/calendar-service.ts, server/oauth-token-manager.ts

Active Self-managed OAuth Secrets: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET

6. Plaid (Financial Data)

Where: Settings page (bank connection), search page (financial queries), morning briefings, personal insights, contextual discoveries.

Why: Connects your bank accounts so Keryx can answer financial questions naturally. Ask "how much did I spend on food this month?" and get a real answer from your actual transaction data. Spending patterns also enrich briefings and insights.

When it activates:

  • When searching: Detects financial queries and routes them to Plaid data instead of memory search
  • Transaction syncing: Periodically pulls new transactions from connected bank accounts
  • Morning briefings: Notable spending included in daily summary
  • Personal insights: Spending patterns analyzed alongside other life data
  • Contextual discoveries: Large transactions can trigger relevant content suggestions

How it works:

  • Uses Plaid API SDK for secure bank connection via Plaid Link widget
  • Flow: Create link token → User authenticates with bank → Exchange public token → Store access token
  • Transaction sync uses Plaid's incremental sync API (cursor-based)
  • Stores items, accounts, and transactions in dedicated database tables
  • Financial queries answered by GPT analyzing actual transaction data
  • Configurable environment: sandbox (testing) or development/production

Key files:

server/plaid-service.ts

Active Secrets: PLAID_CLIENT_ID, PLAID_SECRET, PLAID_ENV

8. Tavily AI Search

Where: Contextual Discoveries page, accessible from the main navigation.

Why: Makes Keryx proactive rather than just reactive. Instead of you searching the web, Keryx analyzes your memories, calendar, emails, and finances to find content that's relevant to YOUR life right now. Planning a trip? It finds travel tips. Mentioned a health concern? It surfaces helpful articles.

When it activates:

  • When you visit the Discoveries page: Generates personalized content based on recent data
  • Analyzes recent memories (last 7 days), upcoming calendar events, email highlights, financial patterns, and location history
  • Results filtered by relevance score (minimum 0.6 threshold)
  • Cached for 30 minutes to avoid excessive API calls

How it works:

  • Step 1: GPT analyzes your recent data to identify "insight contexts" (upcoming trip, recurring topic, notable spending, etc.)
  • Step 2: GPT generates targeted web search queries from those insights
  • Step 3: Tavily executes the searches and returns results with relevance scores
  • Step 4: Results are categorized (travel, shopping, professional, lifestyle, financial, local, general)
  • Urgency levels: immediate, upcoming, or general
  • Notable transaction threshold: $100+ triggers financial discoveries

Key files:

server/contextual-discoveries-service.ts

Active Secrets: TAVILY_API_KEY

9. Location Services

Where: Memory creation (automatic capture), Location History page, Google Timeline import, frequent places detection.

Why: Adds "where" context to your memories. Know not just what you were thinking, but where you were. Builds a history of your movements and identifies places you visit frequently (home, office, gym, etc.).

When it activates:

  • When saving a memory: If location permission is granted, captures GPS coordinates and attaches them
  • Google Timeline import: Bulk import location history from Google Takeout
  • Frequent place detection: Clusters nearby locations to identify places you visit regularly
  • Reverse geocoding: Converts GPS coordinates into readable addresses
  • Location-based reminders: Triggers reminders when you're near a specific place

How it works:

  • Frontend: Uses Browser Geolocation API (navigator.geolocation) for GPS capture
  • Reverse geocoding: OpenStreetMap Nominatim API (free, no API key needed)
  • Rate limited: 1 request per second to comply with Nominatim usage policy
  • Location clustering: Groups locations within proximity radius to detect frequent places
  • Stores history in location_history table, frequent places in frequent_places table
  • Sources tracked: "memory" (automatic capture), "import" (Google Timeline), "manual"

Key files:

server/location-service.ts, client/src/hooks/use-geolocation.tsx

Active No API key needed Browser Permission Required

10. Web Push Notifications

Where: Settings page (enable/disable), background delivery for briefings, alerts, and AI action approvals.

Why: Keeps you informed without needing to open Keryx. Get your morning briefing as a phone notification. Get alerted when the AI detects a pattern. Get prompted to approve an AI action.

When it activates:

  • Morning briefings: Push notification with your daily summary
  • Pattern alerts: When AI detects notable patterns in your data
  • AI action approvals: When an action needs your approval before executing
  • Goal alerts: When AI detects patterns related to your goals
  • Reminder triggers: Time-based reminder notifications

How it works:

  • Uses Web Push API with VAPID (Voluntary Application Server Identification)
  • Service worker (service-worker.js) handles background push events
  • Subscriptions stored per-device in pushSubscriptions table
  • Supports action buttons in notifications (approve/deny for AI actions)
  • Failed subscriptions automatically cleaned up (removes expired/invalid subscriptions)

15. AI Task Execution

Where: Main input (automatic detection), AI Actions page (manage pending/completed actions), Settings (action preferences).

Why: Lets you do things with your voice, not just record things. Say "email Sarah about the meeting" or "remind me to call the dentist tomorrow" and Keryx detects the action, asks for approval, then executes it.

When it activates:

  • Every time you save a memory: GPT scans input for actionable requests
  • Morning briefings, proactive analysis, and automation rules can also generate actions
  • Policy system per action type: Confirm (default — shows pending card), Auto (executes immediately), or Disabled
  • Push notification sent when an action needs approval

All 9 supported action types:

  • calendar.create — Create a calendar event (title, date/time, attendees, location, timezone). Requires Google Calendar or Outlook connected.
  • calendar.delete — Delete an existing calendar event by its event ID. Use when you say "cancel my dentist appointment."
  • email.send — Send a new email via Gmail or Outlook. Requires the respective email integration connected.
  • reminder.create — Set a time-based or location-based reminder. Stored in Keryx and delivered via push notification.
  • people.note — Add a note to a person's contact record (e.g. "note that Sarah said she's moving to Denver").
  • web.search — Search the web via Tavily AI Search and return a summary with sources. Requires TAVILY_API_KEY.
  • memory.create — Explicitly log a new memory entry from an action rather than the main input flow.
  • goal.update — Update the progress percentage on one of your tracked goals (e.g. "mark my fitness goal at 60%").
  • financial.alert — Surface a financial spending pattern or budget concern from your Plaid transaction data.

How it works:

  • GPT analyzes memory text for action intent with structured JSON output
  • Detected actions include: type, title, description, payload, confidence score, reasoning
  • User preference system controls approval policy per action type (Settings → AI Task Execution)
  • Execution routes to the appropriate service (Calendar, Gmail/Outlook, Reminders, Tavily, etc.)
  • Full audit trail: every action logged with status, timestamps, and results
  • Actions have statuses: pending → approved/rejected → executing → completed/failed
  • Rollback available for calendar and email actions after execution
  • Action Chaining: after a chainable action completes, GPT may propose one natural follow-up automatically (see section 23)

Key files:

server/ai-actions-service.ts, server/calendar-service.ts, server/gmail-service.ts

Active Requires: OpenAI + at least one integration

16. Meta Glasses Companion App

Where: Separate React Native app in companion-app/ inside the Keryx repo. Communicates with the Keryx backend via session-authenticated API calls.

Why: Designed for Oakley Meta HSTN glasses. Record memories hands-free while wearing your glasses — no phone needed. Captures GPS location and Bluetooth audio context, and wakes up automatically when you say "Hey Keryx" via Picovoice Porcupine.

When it activates:

  • Always listening (in background) for the "Hey Keryx" wake word via Porcupine SDK
  • On wake word: starts recording speech through Meta glasses Bluetooth SCO audio
  • On recording complete: captures GPS coordinates and sends the full context to POST /api/companion/action
  • Voice queries: say "Hey Keryx, what did I log about the gym last week?" to search memories by voice
  • Status in Settings: the companion card polls GET /api/companion/status and shows a green dot after your first successful recording

How it works:

  • MCP Protocol 2025-01 compliant payloads for structured communication
  • Unified action endpoint: POST /api/companion/action (Life OS tier required)
  • Actions: record (save memory) and query (search memories)
  • Geolocation service: continuous GPS capture enriches every memory with coordinates
  • Bluetooth service: pairs with Meta glasses for SCO audio capture
  • Wake word service: Picovoice Porcupine SDK with custom hey-keryx.ppn model
  • Session authentication: same credentials as the web app — no separate API key needed
  • Connection tracking: backend records last-seen timestamp in memory on every successful action

Setup — step by step

Step 1: Clone and install

git clone https://github.com/merrillnelson-netizen/Keryx.git
cd Keryx/companion-app
npm install

# iOS only (requires macOS + Xcode)
cd ios && pod install && cd ..

Step 2: Create companion-app/.env

PICOVOICE_ACCESS_KEY=<your_picovoice_key>
KERYX_API_URL=https://nelson.replit.app

The KERYX_API_URL is already baked into companion-app/src/services/api.ts as the default, so this variable only needs to be set if you want to point at a local dev server instead.

Step 3: Get Picovoice key + wake word model

  1. Sign up at console.picovoice.ai and copy your free Access Key
  2. Go to the Porcupine Wake Word trainer
  3. Create a model for the phrase "Hey Keryx" — download the .ppn file for Android or iOS
  4. Copy it to companion-app/assets/hey-keryx.ppn
  5. Paste your Access Key into the .env file above

Step 4: Build and install on device

# Android (phone connected via USB or emulator running)
npm run android

# iOS (Mac + Xcode required)
npm run ios

Connect your Meta glasses via Bluetooth before launching — the app will route audio through the glasses automatically.

Step 5: Log in and test

  1. Open the companion app and tap Log In
  2. Enter your Keryx username (email) and password
  3. Say "Hey Keryx, I just had a great run in the park"
  4. The memory is saved to Keryx with GPS context and the green dot appears in Settings

17. High-Signal Alerts

Where: Runs automatically during contextual discoveries, People page (closeness score configuration).

Why: If someone important to you (closeness score 8+) appears in a news article or web result, you want to know about it immediately. This system watches for mentions of your high-priority people in contextual discoveries.

When it activates:

  • Every time contextual discoveries are generated
  • Scans discovery titles and content for mentions of high-priority people (score 8-10)
  • Supports full name matches, first+last name matches, and last name matches with varying confidence levels
  • Skips names that are too short (< 2 characters) to avoid false positives

How it works:

  • Fetches all people with closeness score >= 8 (configurable minimum)
  • Scans discovery text for name matches at three confidence levels: full name (0.95), first+last (0.85), last name only (0.7)
  • Returns matches with context explaining why the alert triggered
  • Results shown as alerts in the discoveries interface

Key files:

server/high-signal-service.ts

Active Depends on: Tavily + People data

18. Database (Neon PostgreSQL)

Where: Everything. Every piece of data in Keryx lives in the database.

Why: The persistent backbone of the entire system. Stores memories, people, settings, AI actions, location history, financial data, messages, ideas, goals, reminders, push subscriptions, and AI cache.

When it activates:

  • Every single API request reads from or writes to the database
  • Session management stored in PostgreSQL
  • Vector embeddings stored for semantic search

How it works:

  • Neon serverless PostgreSQL (managed by Replit)
  • Drizzle ORM for type-safe queries and schema management
  • WebSocket connections via @neondatabase/serverless
  • Schema migrations via npm run db:push (never manual SQL)
  • All queries filter by userId for data isolation
  • Strategic indexes on frequently queried columns
  • JSONB columns for flexible metadata storage
  • Vector type columns for embedding storage (1536 dimensions)

Key tables:

  • users - Account data and authentication
  • log_entries - Memories with AI metadata and embeddings
  • settings - Per-user configuration
  • categories, people - Organization entities
  • aiActions, aiActionPreferences - AI task management
  • ai_cache - Cached AI responses (30-min TTL)
  • location_history, frequent_places - Location data
  • pushSubscriptions - Push notification endpoints
  • ideas, ideaTasks - Workspace items
  • goals, reminders - Tracking systems
  • messageConversations, messages, messageImports - SMS data
  • plaidItems, financialAccounts, financialTransactions - Financial data
  • oauthTokens, oauthNonces - Self-managed OAuth 2.0 token storage (Google + Microsoft)
  • relayEvents, relayDestinations - Universal Relay API fan-out routing and event log

Key files:

shared/schema.ts, server/db.ts, server/storage.ts

Active Env: DATABASE_URL (managed by Replit)

19. Authentication

Where: Login/signup pages, every API route (session validation).

Why: Protects your data. Every piece of information in Keryx is tied to your account and invisible to other users. No one can access your memories, people, finances, or messages without being logged in as you.

When it activates:

  • On every API request: Session cookie validated via Passport.js middleware
  • On login/signup: Password hashed with bcrypt, session created
  • Companion app: Session-based authentication for API calls

How it works:

  • Passport.js with local strategy (username/password)
  • Passwords hashed with bcrypt (10 rounds)
  • Sessions stored in PostgreSQL via connect-pg-simple
  • Session cookie sent with every request for stateful authentication
  • All database queries include userId filter to prevent cross-user data access
  • Object ownership validated before any update/delete operation
  • Zod input validation on all request bodies

Key files:

server/auth.ts

Active No external service

20. Android Bridge

Where: A native Android app (APK downloadable from Settings) that runs as a background service. Pairs with Keryx via Relay API key.

Why: The Android Bridge runs as a native background service and captures Google Messages in real time without any browser involvement. It captures both incoming and outgoing SMS/MMS via the Android content provider, giving you a complete, continuously up-to-date message history without any manual exports.

When it activates:

  • Automatically on device boot (BootReceiver restarts the service)
  • NotificationListenerService captures all incoming messages (SMS, MMS, RCS)
  • SMS ContentObserver foreground service captures sent SMS/MMS via Android content provider
  • Messages are debounced (2.5s buffer) and sent via OkHttp relay client to POST /api/relay/inbound
  • Retries automatically with exponential backoff via WorkManager (6 attempts) when offline

How it works:

  • Native Kotlin app targeting Android 8+ (minSdk 26)
  • Credentials stored securely in EncryptedSharedPreferences
  • Phone numbers normalized to E.164 format via libphonenumber
  • Room database for offline retry queue
  • WorkManager gated on NetworkType.CONNECTED for reliable delivery
  • APK served from Settings page — built via GitHub Actions and linked to the latest GitHub Release
  • Last-seen status displayed in Settings card for connection monitoring

Key files:

android-bridge/ (Kotlin project), client/src/components/settings/android-bridge-card.tsx, server/routes.ts (GET /api/android-bridge/apk-info)

Active Android 8+ (APK) Requires: Relay API key from Settings No external service

21. Ecosystem Dashboard

Where: Accessible from the Dashboard as "Ecosystem View" card. Navigates to /ecosystem route.

Why: Aggregates data from all connected sources — memories, calendar, email, finances, goals, reminders, people, and AI actions — into a single life-at-a-glance dashboard with interactive charts and AI-generated captions. Answers "how is my life going?" at a glance.

When it activates:

  • When you navigate to the Ecosystem page (Pro tier required)
  • Fans out to 10 data sources in parallel via Promise.allSettled
  • Results cached per-user for 5 minutes to avoid redundant AI calls
  • AI generates a contextual caption for each section using GPT

How it works:

  • Backend: GET /api/ecosystem/stats — Pro-tier gated endpoint
  • Mood trend: Area chart with sentiment bands (excellent / good / neutral / challenging)
  • Topic frequency: Horizontal bar chart of top memory topics (last 30 days)
  • Activity: Memories-per-day line chart with daily labels
  • Relationship velocity: Bar chart of people ranked by velocity tier
  • Finance: Spending by category (Life OS tier only, requires Plaid)
  • Goals, reminders, actions: Summary counts with completion rates
  • AI captions via generateEcosystemCaptions() — one sentence per section
  • Charts built with Recharts (AreaChart, BarChart, LineChart)

Key files:

client/src/pages/ecosystem.tsx, server/routes.ts (GET /api/ecosystem/stats), server/ai-service.ts (generateEcosystemCaptions)

Active Pro tier required Requires: OpenAI

22. Automation Rules Engine

Where: Agent page (/agent) → Automations tab. CRUD management via API at GET/POST/PATCH/DELETE /api/automation/rules.

Why: Set it and forget it. Instead of manually tracking life events, you configure rules that fire automatically. "When my mood drops, send me a motivating push notification." "Every morning at 7am, create a goal review reminder." No code required.

When it activates:

  • A trigger event fires anywhere in the app (memory saved, briefing generated, reminder due, etc.)
  • The engine checks all enabled rules matching that trigger
  • Conditions are evaluated against event context
  • If conditions pass and the daily run limit isn't reached, the action executes immediately

10 supported trigger types:

  • memory.logged — fires every time a memory is saved
  • mood.dropped — fires when mood score drops below a threshold
  • mood.spiked — fires when mood score spikes high
  • person.mentioned — fires when a specific person is detected in a memory
  • keyword.detected — fires when a specific word is found in memory text
  • reminder.due — fires when a reminder becomes due
  • briefing.generated — fires after each morning briefing is created
  • goal.updated — fires when a goal's progress changes
  • daily.schedule — fires on a daily cron at a configured hour (0–23)
  • action.completed — fires when a specific AI action type finishes successfully

5 supported action types:

  • send.notification — sends a push notification to all your devices
  • create.reminder — creates a new time-based reminder (supports minutesFromNow or dueAt)
  • create.ai_action — queues a new AI action for approval or auto-execution
  • log.memory — automatically saves a memory with AI-extracted metadata
  • relay.outbound — queues an outbound relay message for a configured destination

How it works:

  • Rules stored in automation_rules table with trigger type, conditions (JSONB), action type, and action payload (JSONB)
  • Condition fields include: moodBelow, moodAbove, keyword, personName, topic, progressAbove/Below, atHour, actionType
  • Action payloads support {{variable}} interpolation (e.g. {{memoryContent}}, {{personName}}, {{goalTitle}})
  • Daily run limit per rule (default: 3). Configurable up to any value.
  • Enable/disable toggle per rule. Test button fires rule immediately for debugging.
  • Execution is non-blocking (setImmediate) — rules never slow down the main request flow
  • Each execution is logged with success/failure status and recorded on the rule's last-run timestamp

23. Action Chaining

Where: Agent page (/agent) → Actions tab. Child actions appear indented under their parent with a chain badge. Toggle: Settings → AI Task Execution → Allow Action Chaining.

Why: Complex tasks involve multiple steps. Instead of you manually chaining "create calendar event → send invite → add contact note," Keryx detects the natural follow-up after each step and proposes it automatically. You stay in control with a single approve/reject per step.

When it activates:

  • After a chainable action completes successfully
  • GPT analyzes the completed action's context and source text for a likely follow-up
  • A follow-up is only proposed if confidence ≥ 0.75
  • Chaining stops at depth 3 (root → child → grandchild maximum)

Chainable action types — both parent and child must be one of these three:

  • calendar.create — may chain to an invite email (email.send) or contact note (people.note)
  • email.send — may chain to a contact note (people.note) or a related email follow-up
  • people.note — may chain to a calendar event (calendar.create) or follow-up email (email.send)

The server enforces a hard whitelist: child action types must also be one of the three chainable types above. Proposals for reminder.create, goal.update, or any other type are rejected at the server level.

How it works:

  • On action completion, maybeSpawnChainedAction() is called with the completed action as context
  • GPT is prompted with the parent action details and source text to detect one natural follow-up
  • The spawned child action records parentActionId (FK → parent) and chainDepth (integer)
  • Child actions inherit the parent's source context in their sourceText field, prefixed with "Chained from: [parent title]"
  • The Agent dashboard shows child actions indented under their parent with a violet left border and a "→ spawned from" badge
  • Toggle: allowActionChaining boolean in the settings table (default: true)

Key files:

server/ai-actions-service.ts (maybeSpawnChainedAction), client/src/pages/agent.tsx (nested chain UI)

Active Controlled by: Settings → Allow Action Chaining

24. Universal Relay API (Inbound & Outbound)

Where: Inbound gateway at POST /api/relay/inbound. Destination management and event log at Agent page → Relay tab. API key managed in Settings.

Why: Acts as a universal message bus between external systems and Keryx. Your Android phone, Meta glasses, SMS gateway, or any custom script can push data into Keryx without a browser session. Keryx can also push data back out — relaying briefings, action results, or custom messages to any configured HTTP endpoint.

Inbound — when it activates:

  • Any system makes a POST to /api/relay/inbound with the X-API-Key header set to your Relay API key
  • The gateway accepts three payload types: sms, command, and event
  • sms — stores the message in the messages system; fan-out to all enabled destinations
  • command — triggers AI action detection on the command text; queues resulting actions for approval
  • event — logs an event to the relay event log; fan-out to configured inbound destinations
  • Every inbound request is logged in the relay event table (visible in the Agent → Relay tab)

Outbound — when it activates:

  • A destination with outboundEnabled = true is configured
  • An AI action of type RELAY_OUTBOUND is approved and executes
  • An automation rule fires the relay.outbound action type
  • The outboundBriefingRelay flag is set — morning briefings are automatically pushed to that destination
  • Outbound format per destination: json (full payload envelope) or text (plain string only)

How it works:

  • Relay API key: generated UUID stored in settings.relayApiKey. Regenerate anytime in Settings.
  • Destinations: stored in relay_destinations table with a URL, name, enabled flag, and outbound config
  • Fan-out: inbound events are dispatched to all enabled destinations via HTTP POST (non-blocking)
  • Retry logic: failed outbound delivery attempts are logged but do not block the inbound response
  • Session-authenticated test endpoint: POST /api/relay/test (for browser-based testing without API key)
  • Event log: all inbound events stored per-user in the relay event table, visible in the Agent → Relay tab
  • Android Bridge uses this API: the Kotlin APK relays Google Messages to /api/relay/inbound in real time

Payload format (inbound):

POST /api/relay/inbound
X-API-Key: <your-relay-api-key>
Content-Type: application/json

{
  "type": "sms",              // "sms" | "command" | "event"
  "source": "android-bridge",
  "payload": {
    "from": "+15551234567",
    "body": "Dinner at 7pm tonight?",
    "timestamp": "2026-04-13T14:32:00Z"
  }
}
